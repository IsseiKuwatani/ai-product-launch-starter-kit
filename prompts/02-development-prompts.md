# AI Webサービス開発プロンプト集 順番に貼る版

Claude Code / Codex に上から順番に貼っていくことで、まず **1画面で動くAI Webサービス風MVP** を作るためのプロンプト集です。

迷ったら飛ばさずに Step 0 から進めてください。最初から高度な機能は作りません。まずは「入力する → 結果が出る → 次に何をすればいいか分かる」状態を完成させます。

## 使い方

1. テンプレートをClaude Code / Codexで開く
2. Step 0から順番に貼る
3. 各Stepの最後に出る「次に貼る引き継ぎメモ」を、次のStepへ貼る
4. Step 4まで終われば、最初のMVPは一度完成
5. Step 5以降は必要な人だけ追加する

## このプロンプト集のゴール

- 誰向けのサービスか分かる
- 入力フォームがある
- 入力内容に応じた結果が表示される
- スマホでも大きく崩れない
- 次に何をすればいいか分かる
- 余計な機能を増やしすぎない

---

## Step 0. 進め方を固定する

最初に一度だけ貼ってください。AIが勝手に大きく作りすぎるのを防ぎます。

```md
あなたは、非エンジニアの事業担当者と一緒に、小さなWebサービスを作るPM兼エンジニアです。

このプロジェクトでは、まず自分のPCで動く1画面MVPを作ります。

ルール:
- まずは「入力する → 結果が出る」体験を完成させる
- 最初はAI接続せず、モック結果で動かす
- ログイン、決済、本格的な保存機能、公開設定は勝手に追加しない
- 1回の変更は小さくする
- 実装前に方針を短く説明する
- 実装後に確認手順を出す
- APIキーや秘密情報はコードに書かない
- 難しい言葉を使いすぎず、初心者にも分かる説明にする

完成条件:
- 入力フォームがある
- 結果が表示される
- スマホでも見やすい
- エラー時に何をすればいいか分かる
- 次に改善する候補が分かる

以後、このルールを守って進めてください。
```

次は **Step 1** に進んでください。

---

## Step 1. アイデアをサービス化する

作りたいものがまだ曖昧な状態で使います。アイデアを「入力 → 結果」の形に整理します。

```md
以下のアイデアを、簡単なWebサービスにできる形へ整理してください。

# アイデア
[ここに作りたいサービスのアイデアを書く]

# 誰向けか
[例: 個人事業主 / 採用担当者 / 店舗オーナー / 学習者]

# 解決したい悩み
[例: 何から始めればいいか分からない / 作業に時間がかかる]

出力してください:
1. サービス名案 5個
2. 一言説明
3. 想定ユーザー
4. ユーザーのBefore / After
5. ユーザーが入力する内容
6. 画面に表示する結果
7. 最初に作る1画面の構成
8. 今回は作らないこと
9. 完成したと言える条件
10. 次のプロンプトに貼るための「引き継ぎメモ」

注意:
- 最初から大きく作りすぎないでください
- 1画面で体験できる形にしてください
- 難しい専門用語は避けてください
```

次は、出力された **引き継ぎメモ** をコピーして **Step 2** に貼ってください。

---

## Step 2. 1画面MVPに絞る

Step 1の内容を、実装しやすい形に絞ります。ここで画面・入力項目・結果表示を決めます。

```md
以下のサービス案を、最初に作る1画面MVPへ絞ってください。

# 引き継ぎメモ
[Step 1の引き継ぎメモをここに貼る]

ルール:
- 1画面で完結させる
- 最初はAI接続しない
- 保存機能は必要最低限にする
- ログイン、決済、本格的な保存機能、公開設定は入れない
- 初心者が見ても分かる言葉にする

出力してください:
1. MVPの目的
2. 画面の流れ
3. メイン見出し
4. 説明文
5. 入力項目
6. ボタン文言
7. 結果表示の構成
8. モック結果の例
9. エラー時の文言
10. スマホで注意する点
11. 次のプロンプトに貼るための「引き継ぎメモ」
```

次は、出力された **引き継ぎメモ** をコピーして **Step 3** に貼ってください。

---

## Step 3. 1画面MVPを実装する

ここで実際にコードを編集します。まずは1画面で動く形を作ります。

```md
このリポジトリで、1画面で完結するWebサービスMVPを作ってください。

# 引き継ぎメモ
[Step 2の引き継ぎメモをここに貼る]

ルール:
- まず1画面で作る
- AI接続はせず、モック結果で動かす
- ログイン、決済、本格的な保存機能、公開設定は追加しない
- スマホでも見やすくする
- 変更は最小限にする
- 実装後に起動確認の手順を出す

実装してください:
1. トップの説明
2. 入力フォーム
3. 送信ボタン
4. 入力内容に応じたモック結果表示
5. 結果の下の「次にやること」
6. 相談や問い合わせにつながる仮ボタン
7. 空欄やエラー時の表示

最後に出力してください:
- 変更したファイル
- 確認手順
- うまく動かない場合に見る場所
- 次の改善候補3つ
```

ここまでで、まず動く形ができます。次は **Step 4** で見た目と使いやすさを整えます。

---

## Step 4. 見た目と使いやすさを整える

Step 3で作った画面を、人に見せやすい状態へ整えます。LP改善・フォーム改善・結果画面改善をここにまとめています。

```md
現在のWebサービスを、ユーザーが使いやすく、価値が伝わる形に改善してください。

改善観点:
- 最初の画面で「誰向けのサービスか」が分かるか
- 何を入力すればいいか迷わないか
- 入力項目が多すぎないか
- ボタン文言が分かりやすいか
- 結果が役に立ちそうに見えるか
- 次に何をすればいいか分かるか
- スマホで見やすいか

実装してください:
1. 見出しと説明文の改善
2. 入力フォームの改善
3. ボタン文言の改善
4. 結果表示の改善
5. エラー文言の追加
6. スマホ表示の調整

ルール:
- 新しい大きな機能は足さない
- まず今ある画面を磨く
- 変更後に確認手順を出す

最後に、完成前チェックに使える「引き継ぎメモ」を出してください。
```

ここまで終われば、最初のMVPは一度完成です。次は **Step 7** で完成前チェックをしてください。保存やAI機能が必要な場合だけ、Step 5 / Step 6 に進みます。

---

## Step 5. 必要なら保存・ダウンロードを追加する

入力内容や結果を残したい場合だけ使います。不要なら飛ばしてください。

```md
現在のWebサービスに、必要最低限の保存・ダウンロード機能を追加してください。

# 追加したいこと
[例: 入力内容を一時保存したい / 結果をファイルとして保存したい]

ルール:
- 個人情報や秘密情報は保存しない
- ユーザー自身のPC内で完結させる
- 不要なら保存機能は追加しない
- 保存していること、削除できることがユーザーに分かるようにする

必要に応じて実装してください:
1. 入力内容の一時保存
2. ページ再読み込み時の復元
3. 保存内容を消すボタン
4. 結果のダウンロード
5. 保存に関する注意書き

最後に出力してください:
- 追加した機能
- 変更したファイル
- 確認手順
- 注意点
```

保存・ダウンロードが確認できたら **Step 7** に進んでください。AI機能も足したい場合だけ **Step 6** に進みます。

---

## Step 6. 必要ならAI機能を追加する

モック結果をAI生成に置き換えたい場合だけ使います。まず設計だけ確認してから、実装に進みます。

```md
現在のモック結果を、AI生成に置き換える設計をしてください。
まだ実装はしないでください。

# 現在の入力項目
[ここに書く]

# 生成したい結果
[ここに書く]

# 期待する出力形式
[箇条書き / 表 / JSONなど]

ルール:
- まず設計だけ出す
- APIキーや秘密情報はコードに書かない
- エラー時の表示を用意する
- 入力が不十分な場合の対応も考える
- コストが増えすぎないようにする
- ユーザーに見せる文章は分かりやすくする

出力してください:
1. どこを変更するか
2. 必要な設定
3. AIに渡す指示文
4. 期待する出力形式
5. エラー時の表示
6. 実装手順
7. 確認方法

最後に「この内容で実装してよいか」を確認してください。
```

設計を確認して問題なければ、次を貼ってください。

```md
上の設計で実装してください。
実装後に、変更したファイル、確認手順、エラー時の確認ポイントを出してください。
```

AI機能を確認できたら **Step 7** に進んでください。

---

## Step 7. 完成前チェックをする

人に見せられる状態か確認します。最後に必ず使ってください。

```md
このMVPが、人に見せられる状態になっているか確認してください。

チェック項目:
1. 起動できるか
2. 何ができるサービスかすぐ分かるか
3. 入力できるか
4. 結果が表示されるか
5. エラー時に迷わないか
6. スマホで見やすいか
7. 余計な機能が入っていないか
8. 秘密情報がコードに入っていないか
9. READMEを見て起動できるか
10. 次にやることが分かるか

出力してください:
- 合格 / 要修正
- 良い点
- 直すべき点
- 優先修正TOP5
- そのまま貼れる修正プロンプト
- READMEに書くべき内容
```

「要修正」が出た場合は、修正プロンプトをそのまま貼って直してください。

---

## 困った時. エラー相談テンプレート

途中でエラーが出た時だけ使います。

```md
以下のエラーを直してください。
いきなり大きく書き換えず、原因を特定してから最小修正してください。

# 起きていること
[例: npm run devでエラーが出る / ボタンを押しても結果が出ない]

# エラーメッセージ
[ここにエラーメッセージをそのまま貼る]

# 直前にやったこと
[ここに書く]

お願い:
1. 原因の候補を3つ出す
2. 一番可能性が高い原因を選ぶ
3. 修正するファイルを示す
4. 最小限の修正をする
5. 確認手順を出す
```

---

## 応用. 1画面で動いた後に3画面へ分ける

最初から使わないでください。Step 7まで終わり、1画面では情報が多すぎると感じた時だけ使います。

```md
現在の1画面MVPを、必要に応じて3画面構成へ分ける案を出してください。
まだ実装はしないでください。

# 現在のMVP
[サービス概要や現在の画面内容を書く]

出力してください:
1. 3画面に分けるべきかどうか
2. 分ける場合の画面構成
3. 各画面の役割
4. ユーザー導線
5. 分けることで良くなる点
6. 分けることで複雑になる点
7. 実装する場合の手順

注意:
- 1画面のままで十分なら、無理に分けないでください
- 実装は、こちらが「実装して」と言ってからにしてください
```
